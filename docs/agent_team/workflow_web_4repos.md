# 多仓 Web 闭环流程（4 库 + 测试）

## 1. 目标

当前阶段只要求先跑通一条“可交付闭环”流程，不追求角色数量和并行度最大化。

闭环定义：

1. 数据采集库
2. 数据公共库
3. 接口库（读取公共库）
4. 前端库
5. 测试环节（作为 terminal gate）

## 2. 仓库职责

1. `collector-repo`
   - 负责采集逻辑与原始数据处理
2. `shared-data-repo`
   - 负责公共数据模型、公共转换逻辑、共享产物
3. `api-repo`
   - 负责服务接口，读取/依赖公共库
4. `frontend-repo`
   - 负责页面、交互、接口消费

每个仓库独立配置（路径、分支、命令、凭据策略）。

## 3. 当前推荐流水线（串行）

```text
collector
-> shared-data
-> api
-> frontend
-> integration-test
-> review/summary
```

说明：

1. 先串行，降低跨仓返工和排障复杂度
2. 跑稳后再评估有限并行（例如 `api/frontend` 的部分并行）

## 4. 最小输入契约（建议）

建议在任务触发时显式提供：

1. `REPO_COLLECTOR`
2. `REPO_SHARED`
3. `REPO_API`
4. `REPO_FE`
5. `BRANCH_COLLECTOR`
6. `BRANCH_SHARED`
7. `BRANCH_API`
8. `BRANCH_FE`
9. `BUILD_CMD_*`
10. `TEST_CMD_*`

最小要求：

1. 所有 repo path 必须是部署机绝对路径
2. 分支策略必须提前明确（新建或复用）
3. 测试 gate 的通过标准必须可机械判断

## 5. 测试闭环定义

测试阶段至少覆盖：

1. 接口可读到公共库产物
2. 前端可消费接口并渲染关键页面
3. 核心路径 smoke/E2E 通过
4. 失败时给出可定位的失败摘要

只有测试 gate 通过，流程才视为完成。

## 6. 当前不纳入范围

本阶段先不做：

1. 多仓并行极限优化
2. 自动发布流水线
3. 复杂审批流
4. 全自动零人工治理

先把“稳定可重复闭环”跑稳，再扩展能力。
