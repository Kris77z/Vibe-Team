# Spacebot-Only 执行基线（重建版）

## 1. 生效结论

生效日期：2026-03-06。

当前阶段只保留 `Spacebot` 作为主链系统，目标是最短路径拿到“可重复交付闭环”。

本阶段不把 `Antfarm`、`OpenClaw` 作为运行前提，不再为三系统联调投入主线精力。

## 2. 阶段目标

1. 建立单系统可运行基线（启动、配置、健康检查、日志）
2. 固化“阶段化 + Gate”交付流程
3. 连续通过 3 次单仓闭环
4. 在稳定后扩展到 4 仓串行闭环

## 3. 工作流模型

统一采用“阶段串行，阶段内可并行”模型。

1. 阶段 A：采集与契约冻结
   - 产出字段、类型、样例、错误码
   - 不允许在后续阶段随意改契约
2. 阶段 B：实现
   - API 按冻结契约实现
   - 前端按契约或 Mock 并行实现
3. 阶段 C：收口
   - 联调
   - 测试
   - 验收

## 4. Gate 定义

1. 采集 Gate
   - 数据正确性通过
   - 边界值与异常值覆盖通过
2. 接口 Gate
   - 契约测试通过
   - 状态码/分页/错误处理符合契约
3. 前端 Gate
   - 关键页面可渲染
   - 核心交互 smoke 通过
4. 集成 Gate
   - 端到端链路通过
   - 失败可定位到明确阶段

## 5. 禁止事项（当前阶段）

1. 以“三系统打通”作为主线成功标准
2. 在未跑稳单仓闭环前投入复杂并行编排
3. 在未定义机械 Gate 前推进多仓集成

## 6. 回切评估条件（仅评估，不默认回切）

以下任一条件持续出现，再评估引入 `Antfarm/OpenClaw`：

1. 并发 workflow 显著上升
2. 明确需要长时间无人值守与自动恢复
3. 多仓编排复杂度超出单系统可维护范围
4. 人工恢复成本高于可接受阈值

## 7. 交付物要求

每次闭环必须产出：

1. 变更摘要（代码/配置/文档）
2. Gate 结果（通过/失败 + 证据）
3. 可复现执行命令
4. 失败定位与下一步动作

## 8. 执行口径

“先稳定，再扩展”。

只要 Gate 还不能稳定复现，就不进入下一层复杂度。
